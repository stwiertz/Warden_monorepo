## Product

Warden is an esports coaching platform for the EVA video game — one product, three deployment surfaces:

- **Mobile app** (`apps/mobile`) — Expo / React Native. Coaches review match footage on-device: import → segment → annotate → export clips. Detection is on-device (FFmpeg, OpenCV via JSI, pHash).
- **Web app** (`apps/web`) — Next.js. Marketing landing + subscription portal. Stripe Checkout & Customer Portal, Firebase Auth, server-side webhook handlers that mirror Stripe state into Firestore.
- **Tooling CLI** (`apps/tooling`) — Python. Validates detection algorithms on desktop and emits the [`map_config.json`](../contracts/map-config.schema.json) consumed by mobile.

## Cross-part integration points

- **`map_config.json` pipeline** — `apps/tooling` generates → [`contracts/map-config.schema.json`](../contracts/map-config.schema.json) is the cross-language source of truth → `apps/mobile` consumes (currently via Firestore `detection_config/latest`, with cache + stale-while-revalidate; see [detectionConfigService.ts](../apps/mobile/src/features/video-processing/detectionConfigService.ts)). Schema validated in Python via `jsonschema` and in TS via Zod (generated by [generate-zod.mjs](../packages/contracts/scripts/generate-zod.mjs)).
- **Firestore `users/{uid}` document** — `apps/web` writes (Stripe webhook handlers in [webhooks.ts](../apps/web/src/lib/stripe/webhooks.ts)), `apps/mobile` reads (subscription gate in [subscriptionService.ts](../apps/mobile/src/features/auth/subscriptionService.ts)). Schema at [`contracts/user-doc.schema.json`](../contracts/user-doc.schema.json) — see [Open Schema Conflict](#open-schema-conflict) below.
- **Shared Firebase project** — both apps point at the same Firebase project (apparent from env conventions but not asserted by code; flagged for Phase 6 confirmation).

## Open Schema Conflict

The `users/{uid}` document is read by mobile and written by web, but the two halves disagree on shape:

- **Web writes** (canonical, Stripe-driven): `{ status, plan, current_period_end, stripe_customer_id, stripe_subscription_id, created_at, updated_at }`. See [subscriptionResponseSchema](../apps/web/src/lib/schemas/subscription.ts).
- **Mobile reads** (legacy denormalized): historically used `isPaid: boolean`. The current code at [subscriptionService.ts](../apps/mobile/src/features/auth/subscriptionService.ts) has been migrated to read `status` + `current_period_end`, but [.env.example](../apps/mobile/.env.example) still references the legacy `isPaid` field, and [user-doc.schema.json](../contracts/user-doc.schema.json) keeps `isPaid` as an optional legacy field.
- **Phase 6 decision pending** — keep `isPaid` as denormalized convenience or derive at read time from `status === 'active' || status === 'trialing'`. The schema's `additionalProperties: true` keeps both halves valid in the meantime.

## Brownfield issues parked for Phase 6

- **Web — Stripe API version drift.** [server.ts](../apps/web/src/lib/stripe/server.ts) pins `STRIPE_API_VERSION = '2026-03-25.dahlia'`; the installed `@stripe` types want `'2026-04-22.dahlia'`. Schema files ([webhook-events.ts](../apps/web/src/lib/schemas/webhook-events.ts)) reflect the dahlia `parent.subscription_details.subscription` nesting. Multiple `*.test.ts` have spread-argument and implicit-any errors (per memory).
- **Mobile — Firebase v12 deprecation.** [firebaseConfig.ts](../apps/mobile/src/features/auth/firebaseConfig.ts) imports `getReactNativePersistence` from `firebase/auth`, which was removed/relocated in Firebase v12. There is also a `// TODO: migrate to @react-native-firebase/*` note in the same file.
- Both Stripe and Firebase issues are **pre-existing** (NOT migration-introduced) — defer until the unified architecture in Phase 6.
