## Successes

- **Single source of truth held across all three stories.** Story 3.1 centralized `src/lib/pricing/plans.ts` with a derived `getYearlySavings()` helper (no literal `"Save €15.98"` string). Story 3.2 extended the same `Plan` type with `stripePriceEnvKey` rather than standing up a parallel Stripe-plan table. Story 3.3 put coupon resolution in one `previewCoupon` helper consumed by the public preview API, the session-create route, AND the SSR pricing page — zero duplication, zero internal HTTP self-calls.
- **Defense-in-depth from Epic 2 reapplied cleanly to payments.** `/api/checkout/session` re-validates coupon codes server-side via `previewCoupon` on every request; client-side `useCheckout()` state is display-only. `STRIPE_SECRET_KEY` and the two `STRIPE_PRICE_*` IDs are server-only, guarded by an extended `.env.example` tripwire test in `src/lib/env.test.ts`. No Price IDs in source.
- **Boundary discipline prevented an Epic 4-shaped race condition.** Story 3.2 explicitly refused to write `users/{uid}` inline on checkout completion — the `users/{uid}` write belongs to Story 4.2's webhook handler. The dashboard success banner copy is deliberately honest: *"It may take a few seconds for your plan to appear."* This avoided the classic "write on return URL, race with webhook" bug class.
- **Stripe SDK mocking pattern delivered (Epic 2 action item #5).** Story 3.2 established `vi.mock('stripe', ...)` with a plain `function Stripe(...)` constructor (the Vitest v4 ES-module-constructor gotcha is captured in 3.2's debug log). Story 3.3 inherited it and extended it for promotion-code lookups without rediscovery. Epic 4 webhook stories can reuse this directly.
- **"Previous Story Learnings" structure held.** Every Epic 3 story carried a concrete Previous Story Learnings block in Dev Notes. Each successive block was *more* concrete than the last — exactly what Epic 2 retro asked for (action item #6). Keep this for Epic 4.
- **Epic 2 carried debt cleared.** `CookieBanner.tsx` `react-hooks/set-state-in-effect` lint error is gone (end-of-epic lint is 0/0, confirmed by both 3.2 and 3.3 completion notes). Epic 1's `Scissors` → `Clapperboard` icon swap is verified in place at [src/app/page.tsx:29](src/app/page.tsx#L29). Both long-carried items are now closed.
- **Stripe mutual-exclusion constraint learned and encoded.** Story 3.3 discovered that Stripe rejects `allow_promotion_codes: true` + `discounts: [...]` simultaneously. The session route handles this by dropping `allow_promotion_codes` only on the coupon path, and a regression test guards the no-coupon path still sends it.
- **SDK shape-drift caught by the type-checker, not production.** Story 3.3 hit the Stripe v22 `PromotionCode.promotion.coupon` nesting change mid-implementation; `npm run build`'s type check caught it before any runtime code shipped. Confirms the project's type-checking discipline is load-bearing.
- **End-to-end flow verified live.** Monthly and yearly checkout both redirect to Stripe, accept `4242 4242 4242 4242`, and return to `/dashboard?checkout=success` with the banner rendering correctly. Story 3.2 Task 8.4 (manual smoke) is now **retroactively cleared**.

## Challenges

- **Silent `catch {}` in a critical API route.** `src/app/api/checkout/session/route.ts` wrapped the Stripe call in a bare `catch {}` with no error logging. When the first live smoke test failed with `CHECKOUT_FAILED`, the dev server console was completely silent — the error was swallowed. Added a `console.error` during this retro to unblock debugging. **This is a defect, not a diagnostic gap:** production observability depends on these errors surfacing somewhere. Every Epic 4 webhook handler will be in the same blast radius.
- **Stripe `prod_` vs `price_` ID confusion — first-time-integration gotcha.** Root populated `.env.local` with Stripe **Product IDs** (`prod_...`) instead of **Price IDs** (`price_...`). The story AC said "Stripe Price IDs" but didn't call out the `prod_` vs `price_` distinction. Stripe's error message (`No such price: 'prod_...'`) was clear — once we could see it. Combined with the silent-catch bug, this wasted a full round-trip of diagnosis.
- **Human prerequisites were invisible to the workflow.** Story 3.2's AC #1 requires a real `STRIPE_SECRET_KEY` + two `STRIPE_PRICE_*` IDs + a test-mode account. There was no point in the story where the dev-story workflow paused and said *"Human: provision these credentials now, here's how."* Task 8.4 quietly downgraded from "E2E verified" to "automated-only, human deferred." Story 3.3 inherited the same deferral silently. This is a systemic workflow gap, not a one-story slip.
- **Stripe CLI PATH not picked up by running VS Code.** After `stripe` CLI install, the existing VS Code integrated terminal did not see it on PATH until VS Code was fully restarted. Wasted one cycle ("command not recognized" in PowerShell). Windows-specific, well-known, easy to avoid once you've hit it once.
- **Hydration mismatch in `PlanCta` — latent bug surfaced by E2E smoke.** Server renders `disabled={true}` (auth resolving, `loading: true`); client hydrates with `disabled={null}`. React warns: `disabled={true}` → `disabled={null}`. The `null` should be a clean `false`. Likely a boolean-coercion bug somewhere in the PlanCta `disabled={loading || pending || ...}` expression. **Unit tests did not catch this because they mock `useAuth` to a resolved state — hydration only mismatches when auth is mid-resolution at hydrate time.** Tests are passing but the warning is live.
- **Vitest parallelism flake.** Story 3.3 hit a global `Cannot read properties of undefined (reading 'config')` error that only reproduced under parallel file execution. Workaround: `npx vitest run --no-file-parallelism`. Root cause not identified. This is a latent test-infra issue that will resurface under load.
- **Stripe webhook signing secret pasted into a chat transcript.** During live debugging, a test-mode `whsec_...` was pasted into this conversation. Low real-world risk (test mode, rotatable), but a process lesson: never paste secrets — even test-mode — into any transcript. Rotated secret is the correct next step.

## Key Insights

1. **A bare `catch {}` is a production defect, not a style choice.** The retro session lost ~20 minutes of diagnosis time because the real Stripe error was visible only to the Node process and then dropped on the floor. Every API route that calls an external service (Stripe, Firebase Admin, webhooks) must log the caught error with enough context to identify the route. This is not optional for Epic 4.
2. **The "Human Prerequisites" gap is structural.** The dev-story workflow assumes the AI agent can do everything, but AC #1 of Story 3.2 literally requires a human to create Stripe accounts, install CLIs, copy IDs. When those steps are invisible in the task list, they get deferred silently and the AC quietly downgrades. This will hit Epic 4 harder than Epic 3 because you cannot unit-test webhook signature verification without a real `stripe listen` session.
3. **Third-party SDK docs deserve the same "read before you code" discipline as Next.js.** AGENTS.md says read Next.js 16 docs before touching App Router APIs. Story 3.3 learned the same lesson applies to the Stripe SDK — the promo-code shape change (`promotion.coupon` nesting) would have been a runtime error if not for the type-checker. Consider adding "read SDK migration notes" to the Dev Notes template for stories that upgrade or adopt a third-party SDK.
4. **Single source of truth scales.** Three stories, one `plans.ts`, one `previewCoupon`, one `CheckoutContext`. Adding coupons in Story 3.3 was small precisely because 3.1 and 3.2 refused to duplicate.
5. **Boundary honesty in UX copy beats boundary ambiguity.** The "It may take a few seconds for your plan to appear" banner acknowledges the webhook gap upfront. Users will accept a few seconds; what they won't accept is a dashboard that says "Active" and isn't. This pattern should carry to Epic 5's dashboard states.
6. **The type-checker is part of CI, and Epic 3 benefited from that.** The Stripe v22 shape drift was caught at `npm run build`, not at runtime. Keep `npm run build` in the story Definition of Done — not just `npm run lint` and `npx vitest run`.

## Team Agreements

- **No silent `catch {}` in any `/api/**` route.** Always `catch (err) { console.error('[route] context:', err); ... }`. Applies retroactively to existing routes (audit in action item #3) and to every Epic 4 handler.
- **Human Prerequisites are checkboxes, not assumptions.** If the dev-story workflow can't execute a step, the story template names it explicitly, and the story cannot move to "done" with that checkbox unchecked.
- **Never paste secrets — even test-mode — into any chat transcript, log, or issue.** Rotate when it happens.
- **`npm run build` is part of story DoD.** Alongside lint and vitest, not instead of a subset.
- **Single source of truth pattern is the template for Epic 4:** webhook payload parsing, event-type dispatch, Firestore writes — all through one typed module with one test file, not scattered across route handlers.
