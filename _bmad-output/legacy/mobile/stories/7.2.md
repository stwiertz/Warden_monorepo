## Story

As a developer,
I want the `ClipExport.view_mode` TypeScript union type to match the new schema (`'full' | 'minimap' | 'minimap_hud'`),
so that the compiler enforces consistency between the data layer and the UI/export logic.

## Acceptance Criteria

1. **Given** the `ClipExport` interface in `src/shared/types/index.ts`, **when** the type is used anywhere in the codebase, **then** `view_mode` has type `"full" | "minimap" | "minimap_hud"`.
2. **Given** any existing code that references `'pov'` as a `view_mode` literal, **when** the type change is applied, **then** the TypeScript compiler reports the error and the code is updated to use `'full'`.
3. **Given** the type is exported, **when** a consumer imports a shared `ViewMode` alias, **then** the alias exists and is re-exported from `src/shared/types/index.ts` for use across features.
4. **Given** the build runs, **when** `tsc --noEmit` is executed, **then** the build succeeds with no errors.

## Tasks / Subtasks

- [ ] Task 1: Update the ClipExport interface (AC: 1, 3)
  - [ ] Edit `src/shared/types/index.ts` — change `view_mode: "pov" | "minimap"` to `view_mode: "full" | "minimap" | "minimap_hud"`
  - [ ] Add and export a named alias: `export type ViewMode = "full" | "minimap" | "minimap_hud"`
  - [ ] Use `ViewMode` as the type of `ClipExport.view_mode` (reference the alias, don't inline)
- [ ] Task 2: Audit the codebase for `'pov'` usages (AC: 2)
  - [ ] Grep for `'pov'` and `"pov"` across `src/`
  - [ ] Replace all string literals with `'full'` where they relate to `view_mode`
  - [ ] Leave unrelated `pov` references untouched (e.g., comments, UI strings)
- [ ] Task 3: Ensure compile passes (AC: 4)
  - [ ] Run `tsc --noEmit` and resolve any remaining errors
