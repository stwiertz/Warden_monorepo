# Story 7.4: Remote Detection Config with MMKV Cache

Status: done

## Story

As a developer,
I want the ROI zones, thresholds, and map hash database to be fetched from Firestore with a local MMKV cache,
so that I can update detection parameters and ship new-map support without releasing a new app version whenever EVA changes its HUD.

## Acceptance Criteria

1. **Given** the app launches for the first time (no cached config), **when** it reaches the home screen, **then** it fetches `detection_config/latest` from Firestore and writes it to MMKV under the key `detection.config`.
2. **Given** the app launches with a cached config, **when** Firestore is reachable, **then** the app fetches the remote `version` field; if remote > cached, the full config is refreshed. Otherwise the cache is used as-is.
3. **Given** the app launches offline, **when** a cached config exists, **then** detection proceeds using the cached values.
4. **Given** the app launches offline with no cache, **when** the coach tries to process a video, **then** a clear error is shown: "Initial setup requires internet — open the app once while online."
5. **Given** the cached config is present, **when** any processing pipeline needs ROI/threshold values, **then** it reads them from a typed accessor (not hardcoded constants).
6. **Given** the config schema, **when** a malformed document is fetched, **then** the fetch is rejected (kept cached config), the failure is logged, and no corrupted config is written.
7. **Given** the `maps` field in the config, **when** map identification runs, **then** it uses the hash entries from the fetched config (not a shipped JSON file, not bundled asset).

## Dev Notes

- **Document path**: `detection_config/latest` (single-document collection). Keep it a single doc rather than a collection to minimize read cost.
- **Admin tooling**: The Python `tools/` pipeline is the upload source. Uploading to Firestore is OUT of scope for this story — manual upload is fine for now. Track automation as a later task.
- **Caching strategy**: Prefer a stale-while-revalidate approach — always read from cache synchronously, refresh in background. This avoids blocking the UI on the network.
- **Storage key**: `detection.config` (nested under an MMKV namespace if the project uses one; follow the convention in `src/shared/services/storage.ts`).
- **Dependency on Story 7.5**: 7.5 replaces the detectors. 7.4 provides the config they read. If 7.5 lands first, it reads from 7.4's config; if 7.4 lands first, existing detectors read their own hardcoded values through the new accessor (no-op read).

### Findings

#### 🔴 HIGH

- **[H1] AC 4 race — bootstrap is fire-and-forget; gate read is synchronous.**
  `App.tsx:50,57` calls `void bootstrapDetectionConfig()` without awaiting. `useVideoProcessing.startProcessing` (line 40) reads `isVideoProcessingBlocked()` synchronously. A user reaching the processing entry point before bootstrap resolves bypasses the offline-first-launch block — the flag isn't set yet, so the gate returns `{blocked: false}` and the pipeline starts. AC 4 specifies "the coach tries to process a video → clear error is shown"; a TOCTOU race weakens this. **Mitigation:** gate the home/import screen on a "bootstrap completed" boolean, or have the entry point call `getDetectionConfig()` synchronously (await the bootstrap promise). The narrow first-launch + offline window is the worst case.

- **[H2] All no-cache fetch errors are mapped to `OfflineFirstLaunchError`.**
  `detectionConfigService.ts:83-90` catches every error in the no-cache path and rethrows `OfflineFirstLaunchError`. If Firestore returns a malformed payload on a fresh install, the user sees "Initial setup requires internet — open the app once while online" while online with a broken remote doc. The bootstrap sets the offline flag; going online won't fix it. AC 4 and AC 6 collide here. **Mitigation:** distinguish `OfflineFirstLaunchError` (network / not-found) from a separate `MalformedRemoteConfigError` (validation throw, document exists). Only the former should set the offline-first-launch flag.

#### 🟡 MEDIUM

- **[M1] Duplicate AC-4 error copy.**
  Defined once in `OfflineFirstLaunchError` (`detectionConfigService.ts:35`) and again as `OFFLINE_FIRST_LAUNCH_MESSAGE` (`detectionConfigBootstrap.ts:21-22`). Single source of truth missing. **Fix:** bootstrap should expose `new OfflineFirstLaunchError().message` (or import the message constant from the service module).

- **[M2] No singleflight on `getDetectionConfig` / `backgroundRefresh`.**
  `detectionConfigService.ts:79` fires `void backgroundRefresh()` per call. Story 7.5's KDA/HSV detector will call `getDetectionConfig` from multiple frames concurrently → N parallel Firestore reads + N cache writes. **Fix:** track an in-flight `Promise<void>` for both `backgroundRefresh` and the no-cache fetch path; return the existing promise instead of starting a new one.

- **[M3] `getCachedDetectionConfig` does not re-validate the cached payload.**
  `detectionConfigService.ts:131-133` returns whatever `JSON.parse` produces, typed as `DetectionConfig`. A schema migration (or corrupted MMKV value from a previous app version) returns an "invalid" DetectionConfig typed as valid → downstream consumers silently dereference `undefined`. **Fix:** wrap the read in a `try { return validateDetectionConfig(cached.config); } catch { return undefined; }` so a stale-schema cache is treated as cache-miss.

#### 🟢 LOW

- **[L1] `validateDetectionConfig` accepts zero-sized ROIs.**
  `detectionConfig.ts:77-81` checks finiteness only. `DEFAULT_DETECTION_CONFIG` relies on this. A mis-uploaded remote with `width:0` passes validation; Story 7.5 consumers crop a 0×0 ROI silently. **Tradeoff:** tightening breaks the bundled default — defer until Story 7.5 populates real ROI defaults.

- **[L2] `getCachedDetectionConfig` reads MMKV + JSON.parse on every call.**
  In Story 7.5's per-frame hot path this is wasteful. **Recommendation for 7.5:** in-memory parsed cache invalidated on `writeCache`.

- **[L3] Module cohesion drift.**
  `isVideoProcessingBlocked` lives in `detectionConfigBootstrap.ts` but is the runtime gate read by `useVideoProcessing`. The bootstrap module mixes a startup-action and a runtime accessor. Move the accessor next to other cache helpers in `detectionConfigService.ts`, or split the module.

- **[L4] `App.tsx` duplication.**
  `void bootstrapDetectionConfig()` appears in both AUTH_BYPASS (line 50) and real-auth (line 57) branches. Hoist after the conditional.

- **[L5] Architecture drift.**
  `architecture.md:403` and `architecture.md:572` name a single `detectionConfig.ts` for "Firestore fetch + MMKV cache". Implementation correctly splits across `detectionConfig.ts` (schema) / `detectionConfigService.ts` (fetch+cache) / `detectionConfigBootstrap.ts` (startup). Update architecture.md to reflect the split.
