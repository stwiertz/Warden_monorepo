# Story 2.2: Integrate FFmpeg and Extract Keyframes

Status: ready-for-dev

## Story

As a developer,
I want a working FFmpeg integration on the Android dev client with reliable keyframe extraction,
so that downstream stories (2.3 black screen detection, 2.4 template matching, 2.5 segmentation, 2.6 background processing) can analyze video frames without decoding the full video and within the 2 GB RAM budget (NFR5).

The FFmpeg surface area for this story exists in code, but it is **currently dead code** — the dependency is not installed and the config plugin is not registered. The service contract (`extractKeyframes`, `extractFrameAt`, `getVideoDuration`, `getProcessingDir`, `cleanupProcessingFiles`) must remain stable because `src/features/video-processing/processingPipeline.ts` already imports it.

## Acceptance Criteria

1. **Given** `package.json`, **when** the developer runs `npm install`, **then** an FFmpeg React Native module is installed whose Android Maven coordinate resolves on a real Gradle build (no `Could not find ...` errors during `expo run:android`). _The architecture's preferred choice is the `jdarshan5/ffmpeg-kit-react-native` community fork, but the `6.0-2` JitPack artifact does not exist — the developer MUST verify the chosen artifact resolves before declaring this AC met. See Dev Notes "FFmpeg dependency resolution" for the decision tree._
2. **Given** `plugins/with-ffmpeg.js`, **when** `expo prebuild --clean` runs, **then** the plugin successfully patches `android/app/build.gradle` with the FFmpeg dependency declaration that matches the artifact installed in AC1, AND `with-ffmpeg` is registered in `app.json` `plugins`.
3. **Given** an MP4 imported via Story 2.1 (`videoImportService.ts`), **when** `extractKeyframes(videoPath, sessionId)` is called, **then** the function returns a non-empty `KeyframeInfo[]` whose `path` files exist on disk and whose `timestampMs` values are monotonically non-decreasing. _Verified by manual smoke test on the dev client; no automated integration test required for this story._
4. **Given** keyframe extraction runs against a ≥30 min H.264/AAC test video on the reference device class (Snapdragon 695 / 6 GB RAM, Poco X5), **then** peak process RAM stays below 2 GB (NFR5). _Verified via Android Studio Profiler or `adb shell dumpsys meminfo` snapshots; record the peak in the Completion Notes._
5. **Given** any consumer of FFmpeg, **then** `src/shared/services/ffmpeg.ts` remains the **sole** entry point — no `require('ffmpeg-kit-react-native')` or equivalent appears anywhere else under `src/`. The exported function signatures (`extractKeyframes`, `extractFrameAt`, `getVideoDuration`, `getProcessingDir`, `cleanupProcessingFiles`) and the `KeyframeInfo` return shape do not change.
6. **Given** the dev client boots after this story lands, **then** the boot-time TypeError documented in commit `0c75f5d` does not return, AND `processingPipeline.runProcessingPipeline(sessionId)` no longer throws "FFmpeg native module not available" when called.

## Dev Notes

### FFmpeg dependency resolution (the hard part)

The architecture document picks `jdarshan5/ffmpeg-kit-react-native` as the MVP fork ([Source: docs/planning-artifacts/architecture.md#FFmpeg Strategy]). The previous Sprint 2 implementation (commit `7eef587`) hardcoded the JitPack coordinate `com.github.jdarshan5:ffmpeg-kit-react-native:6.0-2` in `plugins/with-ffmpeg.js`, but that coordinate does **not resolve** on JitPack — confirmed in commit `0c75f5d` and is the reason the FFmpeg plugin is currently disabled.

**Decision tree for the developer:**

1. **First preference:** install the npm package `jdarshan5/ffmpeg-kit-react-native` (or whatever name it ships under on npm — verify; the GitHub org is `arthenica`'s old fork ecosystem). Inspect its `android/build.gradle` inside `node_modules/` to find the actual `implementation` line it expects. RN autolinking may make the config plugin unnecessary.
2. **Second preference:** if the npm package's autolinking is enough, **delete** `plugins/with-ffmpeg.js` and leave `app.json` `plugins` empty. A no-op plugin is worse than no plugin.
3. **Third preference:** if a manual Gradle line is required, update `with-ffmpeg.js` to inject the actual coordinate the library expects (e.g., the published Maven Central coordinate, not a guessed JitPack one).
4. **If all forks of `ffmpeg-kit-react-native` are dead/unbuildable:** escalate before guessing. The architecture document lists three fallbacks ([Source: docs/planning-artifacts/architecture.md#Risk Alert: ffmpeg-kit-react-native Deprecated]):
   - Option B: Vendor the FFmpegKit AAR locally + custom config plugin
   - Option C: Custom native module via Expo Modules API wrapping FFmpeg
   Both are ≥1 sprint of extra work — flag in Completion Notes and request guidance rather than silently picking one.

**Do not** silently skip the FFmpeg integration or stub it out — Story 2.3 onward depends on real keyframe data.

### Why this story matters more than it looks

The Sprint 2 implementation in commit `7eef587` claimed all of 2.1-2.7 done. In practice:

- 2.1 (video import) genuinely works — no native FFmpeg dependency.
- 2.2 (this story) is dead code — nothing calls into native FFmpeg.
- 2.3 onward are stubs (per the commit message: _"Black screen detection (2.3) and template matching (2.4) are stubbed with proper interfaces — real detection algorithms will be integrated from an external script in Sprint 3"_) but they import from `ffmpeg.ts` and would crash at runtime today.

**So:** until this story closes for real (Android dev client builds AND `extractKeyframes` returns real frames), Sprint 2 is effectively blocked. Treat AC4 (RAM under 2 GB on real device) and AC6 (boot doesn't regress) as gating, not nice-to-have.

### Service contract — do NOT change

Other code already imports from `src/shared/services/ffmpeg.ts`. The contract this story must preserve verbatim:

```ts
// src/shared/services/ffmpeg.ts
export interface ExtractKeyframesOptions {
  width?: number;       // default 320
  quality?: number;     // 1-31, lower=better, default 5
}
export function getProcessingDir(sessionId: string): string;
export function extractKeyframes(
  videoPath: string,
  sessionId: string,
  options?: ExtractKeyframesOptions
): Promise<KeyframeInfo[]>;
export function extractFrameAt(
  videoPath: string,
  timestampMs: number,
  outputPath: string
): Promise<string>;
export function getVideoDuration(videoPath: string): Promise<number>;
export function cleanupProcessingFiles(sessionId: string): void;
```

Where `KeyframeInfo` comes from `src/features/video-processing/types.ts`:

```ts
export interface KeyframeInfo { path: string; timestampMs: number; }
```

If a chosen FFmpeg library forces an API change (different log parsing, different return type), preserve the public surface above by adapting internally — `processingPipeline.ts` and the future detectors should not need to change.

### FFmpeg command pattern (reference)

The current implementation uses:

```
ffmpeg -i {input} -skip_frame nokey -vsync vfr \
       -vf "scale=320:-1,showinfo" -qscale:v 5 \
       {outputDir}/frame_%04d.jpg
```

This is correct and matches the architecture spec. Do not change unless the chosen library requires syntax differences.

Timestamps are extracted from FFmpeg's stderr `pts_time:N.NN` log lines emitted by the `showinfo` filter. The current implementation already has this parser and a directory-listing fallback if log parsing fails — keep both.

### Filesystem API

`expo-file-system@^19` (currently installed) uses the new `Paths`/`Directory` API, NOT the legacy `FileSystem.cacheDirectory` string. The current `ffmpeg.ts` uses `Paths.cache.uri` and `new Directory(path)` — this is correct for v19. Do not regress to the legacy API.

Output directory layout (already implemented, do not change):

```
{Paths.cache.uri}/processing/{sessionId}/keyframes/frame_0001.jpg
                                       /frame_0002.jpg
                                       ...
{Paths.cache.uri}/processing/{sessionId}/results/map_0.jpg     (extracted in Story 2.5)
                                                /map_1.jpg
```

The cache directory is OS-clearable, which is the correct location per architecture: keyframes are recomputable, not user data.

### Testing approach for this story

This story is fundamentally a **native integration** story — its correctness lives in the Gradle build and on a real device. Unit tests against `ffmpeg.ts` would be testing a mocked native module, which has near-zero value. **Skip unit tests for this story.**

What replaces tests:

- Manual smoke test on a real Android device or emulator (Task 4)
- `npm run typecheck` to confirm the public surface didn't drift
- Existing Jest suite (`npm test`) must still pass (currently auth-only)

When Story 2.3 lands, the black-screen detector becomes the natural unit-testable seam — feed it synthetic `KeyframeInfo[]` arrays and assert range output. That's where coverage belongs, not here.

### NFR5 (RAM < 2 GB) — what to actually measure

- The `-vf scale=320:-1` flag is the primary RAM control: low-res frames keep individual decoded buffers small.
- `-skip_frame nokey` ensures only I-frames are decoded, skipping P/B frames entirely.
- FFmpeg natively pipelines frame-by-frame, so RAM should sit well below 2 GB even for 1h20 videos.
- If profiling shows RAM > 2 GB, the most likely culprits are: (a) JS-side accumulating all keyframes in a single array (acceptable — `KeyframeInfo[]` is small), (b) the chosen FFmpeg library leaking session/log buffers across calls (less acceptable — flag and investigate).
