## Story

As a system,
I want to consume detection output from Story 7.5 (game-state transitions + map identification) and combine it into navigable map episodes,
so that the coach can navigate the session as distinct rounds/maps.

## Acceptance Criteria

1. **Given** detection output from Story 7.5 (game-on/game-off START/END events from `gameDetector` + map identifications from `mapIdentifier`), **when** segmentation runs, **then** the system pairs START/END events into game segments and determines start/end time ranges for each map/round (FR7).
2. **Given** segmentation results, **then** lobby and between-round segments (time ranges outside game-on windows) are identified and marked as excluded from navigation (FR8).
3. **Given** segments are determined, **then** each segment is stored in the `map_segments` SQLite table with `session_id`, `map_index`, `start_time_ms`, `end_time_ms`, and `map_name` (from pHash mapIdentifier).
4. **Given** a segment whose `mapIdentifier` returned `null` (no fingerprint within `collision_threshold`), **then** `map_name` is persisted as `null` and the segment remains navigable in Card View.
5. **Given** segments are stored, **then** a result frame path is extracted and stored for each segment (scoreboard screenshot for Card View) -- sampled at `end_ts + score_offset_s` per Story 7.5 AC5.
6. **Given** segmentation completes, **then** the session status is updated to `ready` in SQLite.
7. **Given** the pipeline, **then** it is orchestrated by `src/features/video-processing/processingPipeline.ts`.

## Dev Notes

- **Segmentation algorithm**: This story is a CONSUMER, not a detector. It pairs START/END events from `gameDetector` (Story 7.5) into game segments and overlays `mapIdentifier` results onto each segment. Lobby segments are time ranges outside any game-on window. The detector logic itself (KDA/HSV state machine, long-GOP fallback, pHash matching) lives in Story 7.5 â€” segmentation does not re-implement it.
- **Lobby exclusion**: Time ranges outside START..END pairs from `gameDetector` are lobby/between-round time. Mark these as excluded â€” do NOT create map_segments rows for them.
- **Unknown maps**: When `mapIdentifier` returns `null` for a segment (Hamming distance above threshold for all configured fingerprints), persist `map_name = NULL`. Card View renders these as "Unknown map" â€” they remain navigable.
- **Result frame extraction**: Use FFmpeg to extract a single frame at a specific timestamp: `ffmpeg -ss {timestamp} -i {input} -frames:v 1 -q:v 2 {output}.jpg`. Extract from ~2-5 seconds before the map end (the scoreboard is visible before the black screen).
- **Session status flow**: `importing` â†’ `processing` â†’ `ready` (or `error`). Use existing `updateSessionStatus()` from `sessionRepository.ts`.
- **Database operations**: Use `getDatabase()` from `src/shared/services/database.ts`. Follow existing patterns from `sessionRepository.ts` for INSERT operations.
- **ID generation**: Use `expo-crypto` `randomUUID()` â€” same pattern as `sessionRepository.ts`.
