# Story 7.1: Update clip_exports.view_mode Schema

Status: ready-for-dev

## Story

As a developer,
I want the `clip_exports.view_mode` column to accept three values (`full`, `minimap`, `minimap_hud`),
so that the data model supports the expanded view mode UI (full screen, minimap without HUD, minimap with HUD overlay).

## Acceptance Criteria

1. **Given** the database schema is initialized, **when** the `clip_exports` table is created, **then** the `view_mode` CHECK constraint accepts `'full'`, `'minimap'`, or `'minimap_hud'` (and rejects any other value, including the legacy `'pov'`).
2. **Given** an existing user with `clip_exports` rows containing `view_mode = 'pov'`, **when** the migration runs, **then** those rows are updated to `view_mode = 'full'` (rename, not loss).
3. **Given** a fresh install, **when** the database is opened for the first time, **then** the new schema is applied directly with no migration path executed.
4. **Given** the schema change, **when** any repository query writes to `view_mode`, **then** only the three allowed values can be written (enforced by TypeScript types — see Story 7.2).
5. **Given** the schema migration completes, **then** the schema version in `database.ts` is incremented.

## Tasks / Subtasks

- [ ] Task 1: Update schema definition (AC: 1, 3)
  - [ ] Edit `src/shared/services/database.ts` — change `view_mode TEXT CHECK(view_mode IN ('pov', 'minimap'))` to `view_mode TEXT CHECK(view_mode IN ('full', 'minimap', 'minimap_hud'))`
  - [ ] Bump the schema version constant
- [ ] Task 2: Implement migration for existing users (AC: 2, 5)
  - [ ] Add a migration step: `UPDATE clip_exports SET view_mode = 'full' WHERE view_mode = 'pov'`
  - [ ] Recreate table with the new CHECK constraint (SQLite cannot ALTER CHECK constraints — use the standard rename/copy/drop pattern)
  - [ ] Wrap migration in a transaction
- [ ] Task 3: Update fixtures and tests (AC: 1, 4)
  - [ ] Search the codebase for any hardcoded `'pov'` literal used with `view_mode` and rename to `'full'`
  - [ ] Add a unit test asserting the CHECK constraint rejects invalid values
  - [ ] Add a unit test asserting the migration correctly converts `'pov'` → `'full'`

## Dev Notes

- **SQLite CHECK constraint ALTER**: SQLite does not support `ALTER TABLE ... ALTER CONSTRAINT`. The migration must create a new table with the new CHECK, copy data (translating `pov` → `full`), drop the old table, and rename.
- **Dependency on Story 7.2**: The TypeScript `view_mode` union type must be updated in the same PR to avoid compile errors.
- **Migration safety**: Sprint 2 is already merged. Any sessions processed by alpha testers may have `clip_exports` rows — the migration must not data-loss.

### Project Structure Notes

- Existing schema-version tracking mechanism in that file must be reused — do NOT introduce a new migration framework
