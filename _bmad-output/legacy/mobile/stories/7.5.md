## Story

As a developer,
I want to replace the Sprint 2 black-screen detector and template matcher with a KDA-based Game Detector and a pHash Map Identifier (and keep the black-screen detector only as a fallback for long-GOP videos),
so that round boundaries and maps are detected more reliably using the Warden-tooling methodology while still supporting low-keyframe videos.

## Acceptance Criteria

1. **Given** a video with I-frames every ≤ 2s, **when** the processing pipeline runs, **then** round start/end timestamps are detected by the Game Detector reading the `kda` and `notkda` ROIs (not the black screen).
2. **Given** the Game Detector is analysing a frame, **when** the `kda` ROI contains ≥ 1% white pixels (HSV: `sat ≤ 12` AND `val ≥ 230`) AND the `notkda` ROI grayscale mean is `< hud_brightness_max`, **then** the frame is classified as "in game".
3. **Given** the Game Detector transitions from "not in game" to "in game", **when** `start_confirm_frames` (default 2) consecutive frames confirm, **then** a START is emitted at the first confirmed frame's timestamp.
4. **Given** the Game Detector transitions from "in game" to "not in game", **when** `end_confirm_frames` (default 3) consecutive frames confirm, **then** an END is emitted at the last "in game" frame's timestamp.
5. **Given** an END is emitted, **when** the pipeline computes the score-screen offset, **then** a full-resolution frame is extracted at `end_ts + score_offset_s` (default 14.5) and saved as the `MapSegment.result_frame_path`.
6. **Given** the video has a GOP interval > 2 seconds, **when** the pipeline runs, **then** it switches to the two-pass black-screen fallback: Pass 1 scans `team_bar` saturation on all I-frames to build transition windows; Pass 2 applies the 3-state black-screen detector within those windows.
7. **Given** a segment is detected, **when** the map is identified, **then** a frame mid-segment is sampled, its `map_name` ROI is extracted, grayscaled, resized to 64x64, hashed with pHash (hash_size=8), and compared via Hamming distance to each hash in `DetectionConfig.maps`. The lowest-distance map below the `collision_threshold` is stored in `MapSegment.map_name`.
8. **Given** map identification fails (all distances above threshold), **when** the segment is persisted, **then** `MapSegment.map_name` is `null` and a warning is logged.
9. **Given** the Sprint 2 detectors exist at `src/features/video-processing/blackScreenDetector.ts` and `templateMatcher.ts`, **when** this story is complete, **then** `blackScreenDetector.ts` is refactored to the 3-state fallback pattern, `templateMatcher.ts` is deleted, and `assets/images/map-templates/` is removed.

## Dev Notes

- **Detection methodology reference**: The "Warden-tooling" technical document defines every threshold, ROI, and state transition used in this story. Treat it as authoritative. Any deviation needs justification in the Completion Notes.
- **Thresholds are NOT hardcoded**: All threshold values flow from `DetectionConfig` (Story 7.4). Defaults in the Python reference — `brightness_threshold=15`, `start_confirm_frames=2`, `end_confirm_frames=3`, `sat_max=12`, `val_min=230`, `min_ratio=0.01`, `hud_brightness_max=100`, `score_offset_s=14.5`, `collision_threshold=12` — exist only as documentation of expected remote config values.
- **pHash implementation**: Perceptual hash is small enough (~30 lines) to implement inline. Don't add a new npm dep if `src/shared/services/opencv.ts` already provides DCT primitives or similar. The hash form is 8x8 bits (`hash_size=8`) → stored as 16-char hex.
- **Dependency on Story 7.4**: The detectors need the `DetectionConfig` accessor. If 7.4 ships the no-op shim first, this story replaces the shim's internals.
- **Score screen extraction**: `end_ts + 14.5s` may overshoot the video duration. Clamp to the last available frame, and if the clamp reduces the offset to less than 1s, emit a warning but still save whatever frame is available.
- **ROI scaling**: All ROIs are defined at 1920x1080. At processing resolution (default 720p), scale via the existing `scaleRoi` helper — do not redefine.

### Code Review Fixes — 2026-05-05 (AI)

Adversarial code review pass against the v1 implementation surfaced eleven issues across HIGH / MEDIUM / LOW severities. All except H1 (blocked on `react-native-fast-opencv` native-module wiring) were fixed in code; H1 is tracked under **Review Follow-ups (AI)**.

- **H2 — Streaming detection (memory)**: `processingPipeline.detectGameEvents` no longer eager-loads every keyframe buffer (~600–900 MB for 60–90 min sessions). The shortGop path streams: load → feed FSM → drop. The longGop path runs Pass 1 retaining only one saturation float per keyframe (no buffers), then re-loads only the indices inside high-saturation windows for Pass 2.
- **H3 — Checkpoint semantics**: each pipeline stage now skips when its checkpoint is already `>= stage`, instead of redundantly re-running detection / keyframes when `lastStage === "detection"`. Resume-from-crash now actually resumes instead of restarting from earlier stages.
- **H4 — DEFAULT_DETECTION_CONFIG**: thresholds were placeholder zeroes that would have caused silent total detection failure if the bootstrap gate were ever bypassed. Replaced with the documented Python defaults (`start_confirm_frames=2`, `end_confirm_frames=3`, `sat_max=12`, `val_min=230`, `min_ratio=0.01`, `team_bar_min_sat=25`, `hud_brightness_max=100`, `score_offset_s=14.5`, `collision_threshold=12`). Comment updated to reflect actual behavior.
- **M1 — Decoupled `team_bar_min_sat`**: added a separate threshold to `DetectionConfigThresholds` so the kda white-pixel ratio (`min_ratio`) and the long-GOP team_bar saturation floor are tuned independently. `blackScreenDetector` no longer reuses `min_ratio * 255`.
- **M2 — FSM confirm-gate**: `detectBlackScreensInWindow` no longer fires START on the very first non-black frame after `undetermined`. The seed transition now waits for `start_confirm_frames` consecutive observations like every other transition. New regression test.
- **M3 / L5 — Result-stage error logging**: the bare `try { ... } catch {}` around `extractFrameAt` + `updateResultFramePath` is split and both branches now log the failing segment index, timestamp, and error so prod thumbnail failures are diagnosable instead of presenting as silently-empty Card View tiles.
- **M4 — Resume duration safety**: in the results stage, `videoDurationMs` is re-probed via `getVideoDuration` if both the in-process value and the MMKV `duration` checkpoint are 0. Without this an MMKV cache eviction between stages would skip the score-screen clamp and send a possibly-past-EOF timestamp into FFmpeg.
- **L1 — `scaleRoi` corner rounding**: rounds the right/bottom edges from `(x+width)*sx` so they track the source ROI's edges to within ±0.5 px instead of compounding a separate width rounding on top of an already-rounded x.
- **L2 — Stale comment**: the `DEFAULT_DETECTION_CONFIG` docstring no longer claims behavior it doesn't deliver.
- **L3 — Gitignore**: added `.claude/settings.local.json` (untracked agent-settings file) to `.gitignore`.

**Test impact:** schema gains `team_bar_min_sat`; updated all five DetectionConfig fixtures (`detectionConfig.test.ts`, `detectionConfigService.test.ts`, `gameDetector.test.ts`, `mapIdentifier.test.ts`, `blackScreenDetector.test.ts`) plus `DEFAULT_DETECTION_CONFIG`. Added one new test (`blackScreenDetector` confirm-gate regression). 13 suites / 106 tests pass (was 105).
