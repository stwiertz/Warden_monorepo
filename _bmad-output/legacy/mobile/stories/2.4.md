# Story 2.4: Detect Map End Screens via Template Matching

Status: ready-for-dev

## Story

As a system,
I want to identify map end screens using OpenCV template matching on keyframes,
so that I can precisely determine where each map/round ends.

## Acceptance Criteria

1. **Given** keyframes extracted from a session video (Story 2.2), **when** template matching runs, **then** `react-native-fast-opencv` is integrated via JSI/C++ bindings.
2. **Given** template matching, **then** low-resolution keyframes are compared against map end-screen templates stored in `assets/images/map-templates/`.
3. **Given** a match, **then** matched frames are flagged with their timestamps and confidence scores.
4. **Given** no template matches are found, **then** the system falls back to black screen detection only (FR6).
5. **Given** the integration, **then** the OpenCV service wrapper is at `src/shared/services/opencv.ts` and the template matcher at `src/features/video-processing/templateMatcher.ts`.

## Dev Notes

- **react-native-fast-opencv**: Uses JSI/C++ bindings — no bridge overhead. Shared C++ code works on both Android and iOS. TypeScript API available.
- **Template matching algorithm**: Use `cv.TM_CCOEFF_NORMED` for normalized cross-correlation. This handles brightness variations well. Confidence score is 0.0-1.0.
- **Low resolution**: Both keyframes (from Story 2.2, ~320px wide) and templates should be at matching low resolution. Scale templates down to match keyframe resolution before matching.
- **Fallback design**: This function returns results; it does NOT implement the fallback itself. Story 2.5 (segmentation) decides: if `matchMapEndScreens` returns empty, use `detectBlackScreens` results only.
- **Service boundary**: `src/shared/services/opencv.ts` is the ONLY entry point for OpenCV. The template matcher imports from the service, never from `react-native-fast-opencv` directly.
- **Template images**: For MVP, capture 2-3 Valorant map end screen templates. Name as `{map-name}.jpg` (e.g., `ascent.jpg`, `bind.jpg`). These are static assets bundled with the app.
- **Performance**: Template matching on 320px-wide images is fast (< 10ms per comparison). With ~100 keyframes and ~10 templates, total time is < 10s.
