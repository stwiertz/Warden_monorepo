## Story

As **Stephane** (pre-production, no deployed `MapConfig` consumers on disk or in mobile/web code),
I want **`contracts/map-config.schema.json` rewritten as a single unified shape (no `oneOf`, no v1/v2 branches) keyed by `hud_version` for runtime config selection, with `apps/tooling/tools/map_config_emitter.py` retargeted to read disk-resident zone fragments and emit that shape, and the `jsonschema` strict-validation gate preserved end-to-end**,
so that **the schema mirrors the actual detection pipeline (one detection method, one config shape, multiple HUD versions selected at runtime), the toolchain is O(1) instead of O(branch_count), every future zone consumer (`zone_picker` 9.12, `video_test` 9.13, `roi_detection_tester` refit 9.14, the eventual mobile consumer rewrite) targets one type instead of a discriminated union, and `map_config_emitter.py` decouples from `apps/tooling/config/config.yaml` ahead of Story 9.11's legacy-tooling retirement.**

**Strategic context.** Story 9.9a (merged via `890fa01`) shipped a `oneOf` v1/v2 schema and a dual-emit-path emitter. Post-merge audit confirmed the v1 branch is a **phantom**: no `apps/mobile/assets/map_config.json` exists on disk (Story 1.13 still `backlog`), and `git grep -F MapConfigSchema apps/` returns zero hits — neither mobile nor web imports the generated Zod. The branch design is over-engineered for a pre-production project with zero deployed consumers. 9.9c collapses the branches into one shape. The merged 9.9a work is real history (the `schema_version` field exists, `map_config_emitter.py` exists in its renamed-from-`map_config_generator.py` form, the `jsonschema` gate works); 9.9c rewrites the schema and the emitter on top of that baseline. The full rationale, including the 6-step operator workflow and the rationale for moving fragments off `config.yaml`, lives at [`_bmad-output/sprint-change-proposal-2026-05-15.md`](../sprint-change-proposal-2026-05-15.md).

**Forward absorptions.** Story 9.1 (`schema_version: 1` writer) — already absorbed by 9.9a and already flipped to `cancelled` in the 2026-05-15 correct-course (rationale updated to reflect the 9.9c reframing — `schema_version: 1` is now the config-shape version). The unified emit path preserves `schema_version: 1` as the first dict key. Story 9.4 (jsonschema strict validation gate) — already absorbed by 9.9a and already flipped to `cancelled` in the 2026-05-15 correct-course; the validation mechanism (`_validate_against_schema` helper, `Draft202012Validator`, atomic refusal on failure) is preserved verbatim against the new schema. No additional status flips for 9.1 or 9.4 are needed in this story's post-merge follow-up — they're already done.

**V1 posture.** Out-of-V1-scope at the deliverable level (no mobile consumer yet — that's a separate post-V1 story). V1-safe at the code level: the new schema only ships once Story 1.13 bundles a `map_config.<hud_version>.json` asset. 9.9c does not touch `apps/mobile/` or `apps/web/`.

**Type:** Standard tooling+contracts feature story. Track C (tooling chain). No spike-or-split flag. Mixed Python + JSON Schema + auto-regenerated Zod work; touches `apps/tooling/tools/` (Python emitter rewrite), `apps/tooling/tests/` (pytest rewrite), `contracts/` (JSON Schema rewrite), and `apps/tooling/wardentooling.py` (CLI flag retarget). Single-PR delivery + a tiny post-merge follow-up commit/PR for the sprint-status `review → done` flip + the 9.1/9.4 cancellation flips (Two-PR pattern, [[feedback_two_pr_docs_execution]]).

[x] **AC1 — `contracts/map-config.schema.json` rewritten as a single unified shape.** Top-level `type: object`, `additionalProperties: false`, required fields: `schema_version`, `reference_resolution`, `hud_version`, `score_screen_duration_ms`, `hud_version_detection`, `in_match_detection`, `minimap_identification`. NO `oneOf`. NO `discriminator`. Field definitions:
   - `schema_version: { type: integer, enum: [1] }` — config-shape version (NOT detection-method version, NOT HUD version). Forward-evolution slot; future schema changes bump to `enum: [1, 2, ...]` with a migration story.
   - `reference_resolution: { type: object, required: [width, height], additionalProperties: false, properties: { width: { type: integer, minimum: 1 }, height: { type: integer, minimum: 1 } } }`.
   - `hud_version: { type: string, enum: ["v1", "v2"] }` — HUD generation the config is calibrated for. Runtime selects the matching `map_config.<hud_version>.json` once per session via `hud_version_detection`. New HUD versions extend the enum (small maintenance hook; flag in story 1.13's bundling decision).
   - `score_screen_duration_ms: { type: integer, minimum: 0 }` — timing offset for score-screen extraction after `in_match` falling edge. Per-HUD-version constant; lives in the schema (not the runtime) because it's a config-time tuning knob.
   - `hud_version_detection: { type: array, items: { $ref: "#/$defs/Zone" } }` — zones that classify HUD generation. Evaluated once per session.
   - `in_match_detection: { type: array, items: { $ref: "#/$defs/Zone" } }` — zones that classify "in-match-or-not" (binary). Score-screen is timing-derived (NOT a separate class). Lobby and transition are merged into "not-in-match" (NOT differentiated). This is the **single** game-state classifier — no 4-way `{lobby, in_match, score, transition}` cascade.
   - `minimap_identification: { type: object, required: [id, identification_threshold, roi, maps], additionalProperties: false, properties: { id: <string>, identification_threshold: <number 0.0–1.0>, roi: <Rect>, maps: <object mapping slug → { zones: Zone[] }> } }`. `maps` uses `patternProperties` for `^[a-z][a-z0-9_]*$` slugs, `additionalProperties: false`, `items.$ref: "#/$defs/MapEntry"` where `MapEntry = { type: object, required: [zones], additionalProperties: false, properties: { zones: { type: array, items: { $ref: "#/$defs/Zone" } } } }`.
   - `$defs.Rect`: `{ type: object, required: [name, x, y, width, height], additionalProperties: false, properties: { name: <string min 1>, x: <int ≥ 0>, y: <int ≥ 0>, width: <int ≥ 1>, height: <int ≥ 1> } }`.
   - `$defs.Hsv`: `{ type: object, required: [h_center, h_tol, s_center, s_tol, v_center, v_tol], additionalProperties: false, properties: { h_center: <int 0–360>, h_tol: <int 0–180>, s_center: <int 0–100>, s_tol: <int 0–100>, v_center: <int 0–100>, v_tol: <int 0–100> } }` — **user-space HSV (H 0–360, S/V 0–100), NOT OpenCV-space**.
   - `$defs.Zone`: `{ type: object, required: [id, x, y, width, height, hsv, min_ratio, weight, weight_override], additionalProperties: false, properties: { id: <string min 1>, x: <int ≥ 0>, y: <int ≥ 0>, width: <int ≥ 1>, height: <int ≥ 1>, hsv: { $ref: "#/$defs/Hsv" }, min_ratio: <number 0.0–1.0>, weight: <number ≥ 0.0>, weight_override: { type: ["number", "null"], minimum: 0.0 } } }`. **`weight_override` semantics:** `null` = use `weight` as-is; `number` = override the aggregate weight contribution. This is a behavioral change from V1's `boolean` shape — document in the schema description.
   - Keep `$schema: "https://json-schema.org/draft/2020-12/schema"` and `$id: "https://warden.team/schemas/map-config.json"` unchanged.
   - DELETE the old `$defs`: `V1MapEntry`, `V1ZoneSpec`, `V2ZoneSpecList`, `V2ZoneSpec`. Single `Zone` definition replaces all four.

3. [x] **AC3 — Fragment layout convention.** `map_config_emitter.py` reads exactly 4 files from `<zones_dir>`:
   - `manifest.json` — `{ hud_version: "v1" | "v2", score_screen_duration_ms: integer ≥ 0, reference_resolution: { width: integer, height: integer } }`. These are the per-HUD-version top-level fields that `zone_picker` cannot infer from labeled PNGs alone; the human writes them once per HUD version.
   - `hud_version_detection.json` — JSON array of Zone dicts (matches AC1's `Zone` shape).

## Dev Notes

### Strategic context

9.9c is the cleanup story that retires the v1/v2 schema branching landed by 9.9a. The branching design assumed a `MapConfig` consumer existed that needed v1 to keep working unchanged. Post-9.9a audit confirmed that consumer does not exist: `apps/mobile/assets/map_config.json` is absent from the repo (Story 1.13 owns its creation, still `backlog`); `git grep -F MapConfigSchema apps/` returns zero hits — `mapIdentifier.ts:48` and `gameDetector.ts` run against the handwritten `DetectionConfig` type from `detectionConfig.ts` (Firestore-driven), not the auto-generated Zod. With zero deployed consumers, the `oneOf` branch is over-engineered: it doubles the schema's surface area, doubles `map_config_emitter.py`'s emit paths, and forces every future zone consumer (`zone_picker` 9.12, `video_test` 9.13, the eventual mobile rewrite) to narrow a discriminated union for no operational benefit. 9.9c collapses the shape to one type.

The second motivation is decoupling the emitter from `apps/tooling/config/config.yaml`. The current emitter reads `config.yaml`, detects v1 vs v2 by structural inspection, and emits the matching shape. Story 9.11 will retire the legacy `minimap_identification` block at `config.yaml:87-907` (the v1 input source). If the emitter still depended on `config.yaml` at that point, retiring the block would orphan the emitter. By switching the input layer to disk-resident fragments (`apps/tooling/output/zones/v<hud_version>/`), 9.9c pre-empts the retirement: the emitter no longer needs `config.yaml` at all, so 9.11 can delete the legacy block without cascade work.

Fragment authoring is `zone_picker`'s job (Story 9.12). 9.9c ships the emitter that *reads* fragments; 9.12 ships the tool that *writes* them. In between, fragments can be hand-edited or copied from existing tooling output (e.g., `apps/tooling/output/auto_rois/v2.0/discovered_zones.yaml` — but that's a yaml dump; 9.9c expects JSON fragments). The pytest synthesizes fragments inline; the live-config smoke test from 9.9a is deleted because it no longer makes sense (the live `config.yaml` is the WRONG input shape for the new emitter).

The schema uses `hud_version` as the **runtime selector**, not a schema-level discriminator. A bundled mobile app may carry multiple `map_config.v1.json`, `map_config.v2.json`, etc. (Story 1.13's bundling decision); the runtime classifier (`hud_version_detection` zones) picks one per session. The schema itself describes one config-shape — its instances differ only in which `hud_version` string they carry. This is the right model when "v1 vs v2" is about *which calibration set* applies, not about *which detection algorithm* applies (the algorithm is shared).

### Anti-patterns / disasters to avoid

- **DO NOT keep `oneOf`, `_detect_schema_version`, or any v1/v2 branch dispatch.** Clean cut. If the dev finds themselves writing `if schema_version == 1:` anywhere — STOP. The whole point of this story is single-path.
- **DO NOT preserve `v1` vs `v2` distinction in `schema_version`.** Stephane's posture: `schema_version: 1` is the **config-shape version**, NOT the detection-method version, NOT the HUD version. HUD-version selection moves to the `hud_version` field. If a v2 schema-shape ever ships, it'll be because the *config layout* changed (e.g., adding a new top-level field with no migration path), not because the HUD changed.
- **DO NOT couple to retired tools.** `auto_roi_discoverer/`, `minimap_zone_selector/`, `overlay_stack_analyzer.py`, `frame_labeler.py`, `warden_analyzer.py`, etc. are slated for deletion in Story 9.11. The new emitter does NOT import from any of them. In particular: do NOT import `MAP_LABELS` from `tools.frame_labeler` (9.9a's emitter did; the rewritten emitter must NOT). The new emitter trusts the fragment file's map-iteration order — `zone_picker` writes maps in a stable order; the emitter just preserves it.
- **DO NOT read `apps/tooling/config/config.yaml`.** The emitter is fragment-driven. Story 9.11 may delete chunks of `config.yaml`; the emitter MUST be insulated from that. Remove the `from utils.config import load_config` import — do not replace it with a different config-reading path.
- **DO NOT write a `map_config.json` (no HUD-version suffix).** The output filename is always `map_config.<hud_version>.json`. Story 1.13 will decide how multiple per-HUD-version files are bundled (manifest, picker, etc.) — that's not 9.9c's scope.
- **DO NOT auto-discover `--zones-dir`.** No mtime-based "latest" pick. No filesystem scan. The user passes `--zones-dir` explicitly. The TUI prompt in `flow_tool3` may offer a *menu* of subdirectories under `apps/tooling/output/zones/`, but it does NOT auto-select.
- **DO NOT commit `packages/contracts/src/generated/map-config.ts`.** The path is gitignored per `.gitignore:53`. Apps regenerate at every workspace build via `pnpm --filter @warden/contracts build`. If git stages it, something is wrong with `.gitignore` enforcement — investigate; do NOT force-add.

- **DO NOT touch `apps/mobile/` or `apps/web/` source files.** 9.9c is contracts + tooling only. The mobile consumer rewrite is a separate post-V1 story.
- **DO NOT modify `docs/architecture-tooling.md` or `apps/tooling/README.md`.** Both predate Tools 6/7/8/9 (known doc-debt; deferred docs-refresh). Story 9.10 owns the editorial scrub.
