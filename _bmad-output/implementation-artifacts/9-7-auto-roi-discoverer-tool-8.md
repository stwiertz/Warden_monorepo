# Story 9.7: Auto ROI/HSV Discoverer (Tool 8)

Status: done

## Story

As **Stephane** (sole tooling user; rebuilding the EVA game-state detector for the redesigned HUD),
I want **an interactive review tool that consumes Tool 7's per-`(version, class)` overlay-stack statistics, automatically *suggests* candidate game-state ROI rectangles + HSV bands (the by-eye step `minimap_zone_selector` does by hand — the "autopart"), lets me tweak/accept/reject them and draw exclusion masks, validates each candidate's separability across the four game states, and exports a hand-merge config fragment + a report + per-state previews — without ever touching the live `config/config.yaml`**,
so that **I can turn Tool 7's "average screen" / "what moves" images into concrete, scored `{x, y, w, h, hsv-band}` candidates that the future game-detector re-fingerprinting story can drop into `config/config.yaml`, closing the third and last step of the new-HUD tooling chain (Tool 6 label → Tool 7 stack → Tool 8 discover).**

**Strategic context.** Tool 6 (`video_timeline_labeler.py`, Story 9.5) labels a HUD-version-partitioned PNG dataset; Tool 7 (`overlay_stack_analyzer.py`, Story 9.6) stacks each `(version, class)` cell into `mean.png` (the "average screen — HUD with the game erased") + `stddev.png` (the "mask of every pixel that never moved") + an `overlay_stacks_summary.json`. Today the *next* step — "look at the dark zones of `lobby/stddev.png` and `score/stddev.png`, eyeball where the stable HUD elements sit, pick rectangles, sample their HSV, set tolerances" — is fully manual. `minimap_zone_selector/` does the equivalent for *per-map minimap fingerprints*; Tool 8 does it for *game-state detection* (the `gameDetector` cascade — lobby vs. in-match vs. score vs. transition), where the legacy HUD-1.0 ROIs in `config/config.yaml` (`black_detection.roi_zones`, `minimap_identification`) no longer fire on HUD-2.0 content. Tool 8 automates the candidate-proposal + scoring + validation; the human still reviews, tweaks, and accepts. It produces a **fragment to merge by hand** — it never writes `config/config.yaml`. The actual config update + a `schema_version`-bumped `map_config.json` v2 regen are a separate future re-fingerprinting story, out of scope here.

This story also rides a small **amendment to Tool 7** (Task 1): Tool 7 currently computes HSV-space statistics only under `--heatmap` and even then drops the circular Hue channel. Tool 8 needs (a) the per-pixel HSV stddev (→ how tight an HSV tolerance a pixel can safely carry — *not* recoverable from the `clip(0, 255)` BGR `stddev.png`) and (b) the HSV value at the single most-stable pixel of each cell (the band-center seed). So Task 1 makes Tool 7's HSV-space Welford pass **always-on with proper circular-Hue handling**, persists a per-cell `stats.npz` side-car (`mean_bgr`, `std_bgr`, `mean_hsv`, `std_hsv` float arrays), and adds two fields to each `overlay_stacks_summary.json` `cells[]` entry (`most_stable_hsv`, `stability_percentiles`). The `.png` outputs and `--heatmap` are unchanged as eyeball artifacts; Tool 8 reads the `.npz`. (Alternative considered: split the Tool 7 amendment into its own `9.6.1` micro-story. Rejected — it's ~30 lines of producer-side scaffolding that nothing but Tool 8 consumes; bundling producer+consumer in one PR keeps them in sync and reviewed together. 9.6 is already `done`/merged, so this is a deliberate re-touch, noted in this story's File List; 9.6's status is not re-opened.)

**Type:** Standard tooling feature story. Track C (tooling chain). No spike-or-split flag — bigger than Tool 6/7 individually (GUI + discoverer engine + validator + exclusions + export + the Tool 7 amendment), but Tool 6 (`video_timeline_labeler.py`) set the precedent for a full Tk GUI tool delivered as one story, and the discoverer engine's V1 can be deliberately minimal (a simple stable-region finder + `size × stability × discriminativeness` scoring is enough; fancier segmentation is future polish). Single-PR delivery + a tiny post-merge follow-up commit/PR for the sprint-status `review → done` flip (Two-PR pattern, `feedback_two_pr_docs_execution.md`).

1. [x] **AC1 — Tool 7 amendment: always-on circular-Hue HSV stats + per-cell `stats.npz` side-car + summary fields.** In `apps/tooling/tools/overlay_stack_analyzer.py`:
   - The per-cell streaming processor (`_process_cell`) now **always** runs a second Welford pass in HSV space alongside the BGR pass (drop the `if want_heatmap:` guard on the HSV accumulation). For the **Hue** channel (OpenCV H ∈ 0..179, each unit = 2°) use a **circular** streaming accumulator (running sums of `sin` and `cos` of the per-pixel hue angle), finalized to a circular mean (back in 0..179) and a circular stddev (`R = √(mean_sin² + mean_cos²)`; `std_rad ≈ √(−2·ln max(R, 1e-9))`; convert to H units) — the same math `minimap_zone_selector/app.py:_compute_zone_hsv` uses (lines ~278–284). Saturation and Value use ordinary Welford. Peak RAM stays bounded (a few `float64` images per cell; cells run sequentially) — the HSV pass roughly doubles per-cell work but does not change the memory envelope.
   - For each `status == "ok"` cell, write a side-car `stats.npz` next to `mean.png`/`stddev.png` (`<output>/<version>/<class>/stats.npz`), via `np.savez_compressed`, with arrays: `mean_bgr` (float32, `(h, w, 3)`), `std_bgr` (float32), `mean_hsv` (float32 — channel 0 = circular mean H in OpenCV 0..179 space, channels 1/2 = mean S/V in 0..255), `std_hsv` (float32 — channel 0 = circular std H in H units, channels 1/2 = std S/V), plus a scalar `frame_count` and a 1-D `frame_shape` array. (float32 is sufficient precision and halves disk vs. float64; `savez_compressed` shrinks the smooth mean/std images a lot more.) Use the Windows-non-ASCII-path-safe pattern — write to a `BytesIO` then `Path(dest).write_bytes(...)`, or `np.savez_compressed(file_obj, ...)` — consistent with the tool's existing `imencode` + `Path.write_bytes` PNG writes; never pass a bare path string straight to `np.savez_compressed` if a temp-buffer round-trip avoids the codepage hazard. The per-cell `try/except (OSError, RuntimeError, cv2.error)` already wrapping the PNG writes also covers the `.npz` write (one bad cell → `reason="error"`, batch continues).
   - Each `overlay_stacks_summary.json` `cells[]` `"ok"` entry gains: `most_stable_hsv` (`[h, s, v]` in OpenCV HSV space at the cell's single most-stable pixel — the pixel minimizing the per-pixel **mean of the BGR stddev channels**, the same instability metric AC5's discoverer uses; this is the band-center seed) and `stability_percentiles` (`{"p10": ..., "p50": ..., "p90": ...}` of the per-pixel mean-of-BGR-stddev). Existing fields (`stability_score`, `frame_count`, `frame_shape`, `resized_count`, `outputs`, etc.) are unchanged; `outputs` gains a `"stats"` key (path to `stats.npz` relative to `<output>`, like the existing `mean`/`stddev`/`heatmap` keys). Skipped cells are unchanged (`most_stable_hsv`/`stability_percentiles` absent or `null`).
   - `variance_heatmap.png` stays `--heatmap`-only and may keep its current S+V-only scalar (a proper circular-H term is optional polish, not required — the `.npz` is the real product now).
   - New pure helpers added to the module (testable, no image I/O): a circular-Hue streaming accumulator (init/update/finalize, mirroring `_welford_*`'s shape) and a `stats.npz` build/round-trip helper. `apps/tooling/tools/overlay_stack_analyzer.md` updated: note the always-on HSV stats, the `stats.npz` side-car (contents + that it's the machine-readable companion to the eyeball PNGs), and the two new summary fields.

3. [x] **AC3 — CLI arguments + input loading.**
   - `--input` — Tool 7's output root. Default = the path `overlay_stack_analyzer._default_output_dir()` resolves to (`apps/tooling/output/overlay_stacks`) — import that resolver (or replicate the `__file__`-relative math) so Tool 8's default input always equals Tool 7's default output regardless of checkout location, exactly as Tool 7 did for Tool 6.
   - `--config` — path to `config/config.yaml` (default `config/config.yaml`). **Read-only**: used only to draw the legacy HUD-1.0 ROIs (`black_detection.roi_zones[]`, `minimap_identification.configs[].*.maps.*.zones[]`) as a faded reference overlay on the canvas so the user can see where the old elements sat. Never written. The tool works without it (missing/unparseable → warn, no overlay).
   - `--exclusions` — path to an `exclusions.yaml` (default = `apps/tooling/output/auto_rois/exclusions.yaml` if it exists, else none). See AC4.
   - On startup the loader reads `<input>/overlay_stacks_summary.json` and, for every `status == "ok"` cell, its `stats.npz` side-car. It builds the **four target classes**: `lobby`, `score`, `transition` — each from its own `(version, class)` cell's `.npz` directly — and `in_match` — a **derived/pooled** class combining all the cells whose `class` is one of `frame_labeler.MAP_LABELS` (the 14 canonical maps; cells missing from the dataset are simply not pooled): pooled mean = frame-count-weighted mean-of-means; pooled variance = the parallel/Chan combination of per-cell `(std² · n)` with `n` taken from each cell's `frame_count` in the summary JSON (so `std_bgr`/`std_hsv` for `in_match` are √ of the pooled variance). For the circular-Hue channel, pool by **frame-count-weighted circular mean of the per-cell circular means** (`mean_hsv[..., 0]` — the only H statistic the `stats.npz` carries per AC1): treat each cell's H as an angle, average `sin`/`cos` weighted by `frame_count`, recover the angle. This is a documented V1 approximation — the dev notes it as such in `loader.py` and the `.md`. (A higher-fidelity pool that recombines per-cell `R`/sin-cos sums is out of scope for V1 and would require extending AC1's `stats.npz` schema, which this story does not.)
   - **Shape check — all four target classes must share one `frame_shape`.** Not just the pooled `in_match` map cells: the discoverer (AC5 discriminativeness) and validator (AC6 FP-proxy) compare the `lobby` / `in_match` / `score` / `transition` arrays pixel-for-pixel, so every loaded cell that feeds any target class (the three direct cells *and* every pooled map cell) must agree on `frame_shape`. If they don't (Tool 7 was run without `--ref-height` on a mixed-resolution dataset, so cells carry different modal shapes) → a clean error to stderr listing the disagreeing shapes and telling the user to re-run Tool 7 with `--ref-height` (e.g. `--ref-height 1080`); no traceback. If a `stats.npz` is missing for an `"ok"` cell (older Tool 7 output predating Task 1) → clean message telling the user to re-run Tool 7 to regenerate the side-cars; no traceback. If `overlay_stacks_summary.json` is missing/empty → clean message ("run Tool 7 first"); exit non-zero before Tk loads.
   - All ROI/zone coordinates Tool 8 works in (candidates, exclusions, exported `x/y/w/h`) are in the **cells' pixel space** — i.e. the `--ref-height` resized space if Tool 7 ran with it, otherwise the single common modal shape the loader's shape check verified all cells share. The exported fragment and the `.md` doc must state this coordinate frame explicitly (running Tool 7 with `--ref-height 1080` makes the coords line up directly with `config/config.yaml`'s 1080-reference ROIs — the recommended workflow).

4. [x] **AC4 — Exclusion masks (`exclusions.yaml`).** An optional, hand-editable `exclusions.yaml` carries per-HUD-version → per-target-class lists of *named* rectangles to mask out before discovery — plus an `_all` bucket applied to every class of that version. Shape (YAML):
   ```yaml
   exclusions:
     v2.0:
       _all:
   ```

   Coords are in the cell pixel space (AC3). Pixels inside an excluded rect for a class are never proposed as (part of) a candidate zone for that class (the discoverer treats them as max-instability). The GUI lets the user draw a new exclusion rect on the current class's image, name it (prompt), list/delete existing ones, and **Save exclusions** → writes `exclusions.yaml` back to the `--exclusions` path — or, when `--exclusions` wasn't given (it resolved to `None` because no file existed yet), to `apps/tooling/output/auto_rois/exclusions.yaml` — preserving structure. 5. [x] **AC5 — Discoverer engine (`discoverer.py` — pure helpers, no Tk, no file I/O beyond numpy arrays).** Given a target class's pooled `(mean_bgr, std_bgr, mean_hsv, std_hsv)` arrays, the *other three* classes' `mean_bgr` (and/or `mean_hsv`) arrays, and the class's exclusion mask, `suggest_candidates(...)` returns a ranked list of candidate zones. V1 method (deliberately simple — fancier segmentation is future polish):
   - Compute a per-pixel **instability** map = mean of the BGR stddev channels (lower = more stable HUD chrome). Set excluded pixels to `+inf`.
   - Threshold to a **stable mask** (pixels below a stability cutoff — e.g. the cell's `p10`–`p25` percentile, or an absolute small value; document the choice and make it a tunable constant). Find connected components / contiguous regions in the stable mask; for each region above a minimum area, take its axis-aligned bounding box → a candidate rect (optionally shrunk to the rect's stable sub-core).
   - For each candidate rect, derive an **HSV band**: `h_center` = the rect's circular-mean H, `s_center`/`v_center` = mean S/V over the rect (all from `mean_hsv`); `h_tol` = `k · (rect's H circular std)`, `s_tol`/`v_tol` = `k · (rect's S/V std)` over `std_hsv`, each clamped to a sensible minimum (reuse `minimap_zone_selector`'s `_MIN_H_TOL = 10`, `_MIN_SV_TOL = 5` and a `k ≈ 1.5` factor — same as `_compute_zone_hsv`). Convert centers/tolerances to **user space** (H 0–360, S/V 0–100) for the exported fragment; keep OpenCV-space internally where convenient. Set `min_ratio` to a default (`0.3`, matching `minimap_zone_selector`).
   - **Score** each candidate = `f(size, stability, discriminativeness)` where: *size* = a (clamped) function of rect area (bigger is more robust, with diminishing returns / a cap so a giant low-info rect doesn't dominate); *stability* = inverse of the rect's mean instability (tighter band possible); *discriminativeness* = the HSV distance between this rect's `mean_hsv` in the target class vs. the same rect in each of the other three classes' `mean_hsv` — take the **minimum** over the three others (a zone is only as discriminative as its worst confuser), distance computed with circular handling on H (e.g. weighted Euclidean over `(Δh_circular, Δs, Δv)`). Document the exact scoring formula in code and in the `.md`. Return candidates sorted descending by score, each carrying its rect, HSV band, the three component sub-scores, and which "other class" was its closest confuser.

6. [x] **AC6 — `GameStateValidator` (`validator.py` — pure, no Tk).** Distinct from `minimap_zone_selector`'s map-keyed `ZoneValidator`: this one is **game-state-keyed** over the four classes. Given a set of accepted candidate zones each assigned to a target class, and the per-class `mean_bgr`/`mean_hsv` (and `std_*`), compute a **separability/confidence proxy** per zone and per class:
   - per-zone **TP-proxy**: does the zone's HSV band fire on its assigned class's `mean_bgr` (apply the band like `zone_model.zone_fires`'s `cv2.inRange` + `min_ratio` test, but against the mean image), and an estimate of the fraction of *individual* frames it would fire on, derived from how far the band edges sit from the class mean relative to that region's `std` (a wider margin in std-units → higher coverage estimate).
   - per-zone **FP-proxy**: does the band fire on each of the other three classes' `mean_bgr`; the worst (max) is the FP rate.
   - per-class **separability**: with all of that class's accepted zones combined under a weighted/`identification_threshold`-style rule (reuse `minimap_zone_selector`'s 0.6-threshold + per-zone-weight idea, or a simpler "≥1 zone with TP-proxy ≥ X and FP-proxy ≤ Y" rule — document it), does the class come out cleanly separable from the other three? Surface a per-class verdict + the contributing zones.
   - **Explicitly documented as a proxy** — it's mean-image-/std-based, not per-frame. Exact per-frame validation against the labeled PNG dataset (re-streaming Tool 6's frames through the band) is **out of scope** here and is part of the future re-fingerprinting story; the `.md` says so. (Rationale: Tool 8 reads Tool 7's aggregates by design — loading hundreds of 1080p frames per class would defeat the point.)
   - `transition` is handled like every other class — its zones (if any survive) tend to be low-discriminativeness by nature; the engine ranks them last on their own merits; **no `transition` special-casing in the engine** (consistent with Tool 7's "stack whatever folders it finds" stance). The point of keeping `transition` as a target class is that the future detector cascade must *recognize and reject* transition frames, not key game-state detection on them.

## Dev Notes

### Strategic context

Tool 8 is the **third and last link of the new-HUD tooling chain**: Tool 6 (`video_timeline_labeler.py`, Story 9.5) produces a HUD-version-partitioned labeled PNG dataset; Tool 7 (`overlay_stack_analyzer.py`, Story 9.6) stacks each `(version, class)` cell into `mean.png`/`stddev.png` + `overlay_stacks_summary.json`; Tool 8 (this story) consumes Tool 7's per-cell statistics and **automates the by-eye ROI/HSV pick** — proposing candidate game-state zones, scoring them by size × stability × discriminativeness, validating their separability across the four game states, and exporting a hand-merge config fragment + report + previews. The legacy `config/config.yaml` ROIs (`black_detection.roi_zones`, `minimap_identification`) target the legacy HUD 1.0 and no longer fire on HUD 2.0; Tool 8's output is the raw material for the future game-detector re-fingerprinting story that updates `config/config.yaml` and regenerates `map_config.json` v2 — **that update is out of this story's scope**. Tool 8 only ever *reads* `config/config.yaml` (to show the legacy ROIs as a reference overlay); it never writes it.

The four target classes and why: `lobby` and `score` each have their own static chrome (lobby UI; the scoreboard panel/grid/headers — invariant even though the scoreboard *backdrop* varies per-map and per-instance, which is why `score/` stays a flat bucket and stacking it surfaces exactly the invariant scoreboard); `in_match` is a **derived/pooled** class — the 14 `MAP_LABELS` cells combined (frame-count-weighted mean-of-means; Chan/parallel variance pooling) so the in-match HUD chrome (which is map-independent) comes out sharp while the per-map scenery averages to haze; `transition` is kept as a target class so the future cascade *recognizes and rejects* transition frames — its stacked output is incoherent by design (it lumps black screens, lobby-fading-to-black, countdown overlays, map-fading-in), so Tool 8 ranks its candidates last on their own (low-discriminativeness) merits — **no `transition` special-casing in the engine**, consistent with Tool 7's "stack whatever folders it finds" stance.

### Anti-patterns / disasters to avoid

- **DO NOT auto-edit `config/config.yaml`.** Tool 8 *reads* it (legacy-ROI reference overlay) and *exports a hand-merge fragment* (`output/auto_rois/<version>/discovered_zones.{json,yaml}`). It never writes `config/config.yaml`. The hand-merge step is the human's, in the future re-fingerprinting story. (Contrast `minimap_zone_selector`, which *does* write `config.yaml` for `minimap_identification` — Tool 8 deliberately does not, both because its target is the not-yet-defined game-state config section and because the export-then-review-then-merge separation is the point.)
- **DO NOT re-stream Tool 6's labeled PNGs.** Tool 8 reads Tool 7's aggregates (`stats.npz` + summary JSON) — that's why the Tool 7 amendment exists. Loading hundreds of 1080p frames per class would defeat the architecture; the `GameStateValidator` is a mean/std *proxy* by design.
- **DO NOT special-case `transition`.** It's a regular target class; its candidates score low on discriminativeness on their own merits and the engine ranks them accordingly. Keeping it as a class is so the future cascade *recognizes and rejects* transition frames — not so we key detection on transition pixels.
- **DO NOT `np.stack` all frames in Tool 7 (Task 1).** The amendment adds a second (HSV) Welford pass — still streaming, still a few `float64` images per cell, cells sequential. The `stats.npz` is float32-compressed precisely so the always-on cost (disk + time) stays modest on the 16-cell / 2,748-frame real dataset.
- **DO NOT use `cv2.imread`/`cv2.imwrite` with bare paths, or pass bare path strings to `np.savez_compressed`.** Windows non-ASCII path hazard (the Tool 6/7 reviews caught the `imread`/`imwrite` side). Use `np.fromfile`+`imdecode` / `imencode`+`Path.write_bytes` for PNGs and a `BytesIO` round-trip for `.npz`.
- **DO NOT create multiple `tk.Tk()` roots in the GUL.** Single root; child dialogs as `Toplevel`/`messagebox`/`simpledialog`. (Tool 6's review flagged the multi-root hazard.)
- **DO NOT crash on a missing/old/partial input.** Missing `overlay_stacks_summary.json` → "run Tool 7 first"; missing `stats.npz` on an `"ok"` cell → "re-run Tool 7" (output predates Task 1); mixed-shape pooled cells → "re-run Tool 7 with `--ref-height`". Clean stderr message, exit non-zero *before* Tk loads; never a bare traceback. Parse errors in `config.yaml` or `exclusions.yaml` → warn + degrade gracefully.
- **DO NOT silently divide by zero** in the band-derivation (constant region → use the `_MIN_*_TOL` floors), the Chan pooling (a single-frame map cell → variance term is 0, fine), or any normalization.
- **DO NOT modify `docs/architecture-tooling.md` or `apps/tooling/README.md`** — both predate Tools 6/7/8; add the sibling `auto_roi_discoverer.md` (Story 9.5/9.6 precedent). A docs-refresh story will catch the as-built docs up later.
