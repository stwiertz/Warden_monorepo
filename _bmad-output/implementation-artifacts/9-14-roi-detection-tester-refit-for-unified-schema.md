## Dev Notes

### Strategic context — what this story is and is NOT

- **This is a refit of one existing single-file headless tool, not a new tool.** `apps/tooling/tools/roi_detection_tester.py` (1171 lines, was Story 9.8 — DONE) + its test file (696 lines) are the deliverable surface. The structure (loader → `iter_labeled_frames` → `evaluate_frame` → `aggregate_metrics` → `write_report`/`write_summary`/`write_frame_predictions` → `main`) is sound and stays — what changes is the *input shape* (Tool 8 yaml fragment → unified `map_config.json`), the *classifier set* (4-way game-state cascade DELETED → HUD-version NEW + binary in_match), and the *report structure* (2 blocks → 3 classifier sections).
- **The single largest change is the input contract (AC0).** Today Tool 9 reads `discovered_zones.yaml` from the retired Tool 8. That file format no longer exists post-9.11. The story rebases the loader onto the Story 9.9c unified `map_config.<hud_version>.json` — the actual shipping artifact `apps/mobile` will eventually consume, and the artifact `video_test` (9.13) also consumes. Reading the post-emit, jsonschema-validated config (not raw fragments) means Tool 9 validates the *same bytes the runtime sees*.
- **Why the emitted config, not fragments:** The 9.9b loop is literally `zone_picker → map_config_emitter → video_test and/or roi_detection_tester → adjust → repeat` (sprint-change-proposal-2026-05-15.md §9.9b). Tool 9 sits *downstream of the emitter*. 9.13 video_test "consumes an emitted `map_config.<hud_version>.json`". For Tool 9 and video_test to validate the same artifact (so labeled-PNG accuracy and video accuracy are comparable), Tool 9 must read the emitted config. "Refit for the unified schema" literally means: conform to `contracts/map-config.schema.json`, whose only instance is `map_config.<hud_version>.json`.
- **Per-map scoring: weighted vs "unchanged" (AC6 / AC0a).** AC1 of the proposal says per-map ID is "(UNCHANGED)". Read narrowly that means *keep the unweighted `fires/n_zones` ratio*. But the unified schema added `weight`/`weight_override` and `minimap_identification.identification_threshold` whose own description is "Minimum aggregate match score (sum of per-zone fires × weight)", and Story 9.12 added per-zone weight editing specifically for "100%-accurate per-map ID". A validator that ignores weights would measure something the runtime does not do — defeating the point of step #5. Recommendation: **weighted aggregate** (the faithful reading; "UNCHANGED" = the classifier's *role*, not its arithmetic, since the Zone shape itself changed). Confirm at kickoff (AC0a). If literal-unchanged is chosen instead, `weight`/`weight_override` are still parsed (schema-required) but unused by the scorer, and `identification_threshold` is replaced by `--map-threshold` on the `fires/n` ratio.
- **HUD-version classifier shape.** A single `map_config.<hud>.json` carries exactly one HUD's `hud_version_detection` zones. So the classifier is structurally a *per-HUD binary detector* ("is this frame this HUD's HUD?") evaluated against multi-HUD ground truth — the exact same pattern as the binary in_match classifier (one positive class + `unknown`/negative). Do not try to make it N-way across HUD versions from one config; that only becomes N-way when multiple configs are bundled at runtime (Story 1.13 territory, out of scope here). Confusion columns = `{<config.hud_version>, unknown}`.
