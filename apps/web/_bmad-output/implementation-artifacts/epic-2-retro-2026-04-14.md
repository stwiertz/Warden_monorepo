## Successes

- **100% story completion** — all 4 stories delivered, no descoping, no carryover
- **Compounding reuse pattern** — Story 2.1's primitives (`createSessionAndRedirect`, spinner markup, `role="alert"` error display, Firebase error → message map) were explicitly reused and referenced in every subsequent story. Each story's "Previous Story Learnings" section grew more concrete and load-bearing.
- **Symmetry discipline** — Story 2.3 added `destroySessionAndRedirect` mirroring the existing create helper, and refactored the duplicated dashboard sign-out in the same PR. No duplication debt left behind.
- **Defense-in-depth auth architecture (Story 2.4):** three independent layers
  - **Proxy** (Edge runtime) — cookie presence check only, no `firebase-admin` import
  - **RSC layout** (Node runtime) — `adminAuth.verifySessionCookie(cookie, true)` with revocation check, authoritative gate
  - **API routes** (Node runtime) — same full verification via `requireSession()`, no trust inherited from proxy
- **Test hygiene held.** Epic 1 retro flagged fake assertions; Epic 2 had zero. Every story explicitly verified "zero regressions" against a named baseline (52 → 72 → 101 → 119 → 153).
- **Review culture caught real bugs, not nits.** Most significant saves:
  - Story 2.3: `signOut()` was called before DELETE; if `signOut` threw, the HttpOnly session cookie would have lived on the server. Fix wraps `signOut` in try/catch and always issues the DELETE.
  - Story 2.4 M1: brittle `err.message.includes('revoked')` error classification replaced with Firebase Admin's typed `err.code` discriminator (`auth/session-cookie-revoked` / `auth/session-cookie-expired` / `auth/id-token-expired`).
  - Story 2.4 M2: intended-destination preservation through the RSC layout via a proxy-set `x-warden-pathname` request header, so `/dashboard/settings?tab=billing` with an expired cookie now routes back to the exact path, not `/dashboard`.
- **Open-redirect defense implemented correctly** — `sanitizeRedirect` util rejects `//evil.com`, `/\evil.com`, `https://...`, null, and empty; tested with explicit attack inputs.
- **Security contracts encoded as tripwires** — `src/lib/env.test.ts` asserts `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `FIREBASE_SERVICE_ACCOUNT_KEY` are never `NEXT_PUBLIC_`-prefixed in `.env.example`. Future regressions fail a test, not a prod postmortem.
- **Pattern continuity from Epic 1 retro held** — no fake test assertions, config paths resolve correctly, design system reuse continued.

- **Manual Firebase-auth tests remain unchecked.** Stories 2.2 marked manual test steps 7.4–7.7 incomplete (require real Firebase creds). Automated test coverage is strong, but live end-to-end verification has not been run.
- **Firestore rules committed but not deployed.** `firestore.rules` and `firebase.json` landed in Story 2.4, but the manual `firebase deploy --only firestore:rules` step has not been executed. FR32 is not enforced in production yet. This becomes blocking before Epic 4's webhook handlers start writing `users/{uid}` docs.
- **Carried debt: `CookieBanner.tsx` lint error** — pre-existing `react-hooks/set-state-in-effect` from Epic 1, explicitly marked "out of scope" in Stories 2.3 and 2.4. The exemption is now a pattern, which is a smell.

## Key Insights

1. **Story-to-story learning propagation works when it's structured.** Every Epic 2 story had a "Previous Story Learnings" block in its Dev Notes. This forced Amelia to retrieve and apply prior gotchas instead of rediscovering them. Keep this structure for Epic 3+.
2. **AI code review remains load-bearing.** The Story 2.3 signOut-before-DELETE bug and the Story 2.4 error-classification bug are exactly the kind of subtle auth issues that pass unit tests but break security properties. Do not skip review on Stripe/webhook stories.
3. **Pattern symmetry is a refactor forcing function.** "If you add the destroy helper, you also replace the duplicated dashboard sign-out" — made the refactor non-optional and eliminated duplication debt at the moment it was introduced.
4. **Defense-in-depth is cheap when the layers have clear ownership.** Proxy = presence (fast, Edge-safe). RSC layout = authoritative (Node, revocation check). API routes = independent verification (Node, no trust inherited). Each layer's purpose was named in Story 2.4's Dev Notes, making the architecture review-able.
5. **Tripwire tests beat documentation for security contracts.** `env.test.ts` is a one-time investment that outlives any README warning.
6. **"Out of scope" is a habit that compounds.** Two stories in a row deferred the `CookieBanner` lint error. Three would be a policy. Fix it before Epic 3.

## Team Agreements

- Code review remains mandatory, especially for any story touching auth, Firestore writes, or webhook handlers.
- Defense-in-depth pattern (proxy presence + RSC authoritative + API independent) is the template for Epic 4 webhooks: proxy must not trust payload, route handler must independently verify Stripe signature.
- `requireSession()` / `withAuth()` from `src/lib/firebase/auth.ts` is the **mandatory** pattern for every future `/api/subscription/*` and `/api/checkout/*` route. No inline cookie reads.
- `sanitizeRedirect` must be applied to every `next=` param consumer. Do not hand-roll open-redirect defense.
- "Out of scope" exemptions on lint/test errors require an explicit follow-up action item in the same PR; no silent deferral.
