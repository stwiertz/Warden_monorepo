## Successes

- **"Thin HTTP shell + logic in lib/" architecture scaled perfectly.** Story 4.1 established the split: `src/app/api/webhooks/stripe/route.ts` is a thin HTTP shell (signature verification, dedup transaction, response envelope), and `src/lib/stripe/webhooks.ts` contains all routing and handler logic. After 4.1 landed, **neither Story 4.2 nor 4.3 ever touched `route.ts` again**. This is the Single-Source-of-Truth pattern Epic 3 retro validated, applied at the architecture level.
- **Zod schema defense-in-depth validated on real API drift.** Story 4.2's manual smoke discovered that Stripe API version `2026-03-25.dahlia` had relocated `invoice.subscription` to `invoice.parent.subscription_details.subscription`. The `invoicePaidSchema` Zod validation caught it on the first live event — failed fast, logged clearly, returned 200 to Stripe, and the event stayed in `stripe_events` for replay after the 3-file fix. Story 4.3's `paymentFailedSchema` inherited the correct nested path.
- **Human Prerequisites block worked as designed.** All three stories used the Epic 3 retro action item #7 format. No silent deferrals, no quietly downgraded ACs. Every manual smoke step was completed by Root before the story moved to done.
- **Previous Story Learnings / Dev Notes structure produced rich knowledge transfer.** Each story's Dev Notes carried concrete, actionable intelligence from prior stories — the dahlia fix, the `import * as self` pattern, the plain-function Stripe constructor mock, the silent-catch discipline. This prevented rediscovery of solved problems.
- **`import * as self from './webhooks'` self-namespace pattern held across all 3 stories.** Established in Story 4.1 for `vi.spyOn` interception of intra-module calls, confirmed working through Story 4.2's real handler implementation, and confirmed again in Story 4.3 with two more real handlers. The dispatch tests in `webhooks.test.ts` never needed modification.
- **Silent-catch discipline enforced (Epic 3 retro #3 payoff).** Every `catch` in Epic 4 has a `console.error` with a `[webhooks/stripe]` prefix before acting. The audit grep `rg -n 'catch \{' src/app/api/webhooks src/lib/stripe` returns zero silent catches. This directly saved debugging time — every error was visible in the dev-server console during manual smoke testing.
- **`retryStripeCall` helper built once, reused cleanly.** Story 4.2 Task 2 added the 3-attempt retry helper to `server.ts`. Story 4.3's `handlePaymentFailed` reused it without modification. No copy-paste, no npm library, no scope creep.
- **`didWrite` gating pattern in Story 4.3.** Both terminal handlers use a simple `let didWrite = false` above the transaction, flipped to `true` before `tx.update`. Post-transaction code gates the success log on `if (didWrite)`. Clean, readable, no clever abstractions.
- **`tx.create` / `tx.update` invariants held.** Never `tx.set` anywhere in Epic 4. The invariant (create throws on exists, update throws on not-exists) caught the right failure modes and kept the error signal clean.
- **`npm run build` as separate DoD line (Epic 3 retro #10).** Every story had a distinct build task. Story 4.2's dahlia drift was fixable precisely because the build/type-check step is explicit.

## Challenges

- **Stripe dahlia API drift in Story 4.2.** The AC #3 schema block was drafted against the pre-dahlia `invoice.subscription` shape. The pinned API version `2026-03-25.dahlia` had already relocated it to `invoice.parent.subscription_details.subscription`. All unit tests passed because hand-crafted fixtures matched the old shape. It took a live `stripe trigger` event during Task 10 manual smoke to surface the mismatch. **Root cause:** story AC field paths were not verified against the actual pinned SDK types during drafting. **Impact:** one debugging cycle + 3-file fix. **Mitigation going forward:** verify AC field paths against `node_modules/stripe/types/*.d.ts` before finalizing stories.
- **Manual smoke instructions lacked "where to look" clarity.** Root found the manual testing steps sometimes unclear about where to put or read things — which terminal, which console, which UI, which field. The Human Prerequisites block was a big improvement over Epic 3's invisible steps, but the verification steps could be more explicit about navigation paths (e.g., "Firebase Console → Firestore Database → `users` collection → document with your uid → `status` field").
- **Vitest file-parallelism flake still unresolved.** Epic 3 retro action item #6 (in progress). The `Cannot read properties of undefined (reading 'config')` crash on `dashboard/layout.test.tsx` resurfaced in Story 4.2 under parallel execution. Workaround `--no-file-parallelism` used. Root cause still unidentified. Now at 37 test files and growing.
- **Epic 3 retro items #1, #2, #4 were carried without resolution.** Firestore rules deploy, Firebase auth E2E, and PlanCta hydration fix were all deferred through Epic 4. Resolved during this retro by planning them into a launch-readiness epic rather than continuing to carry them.

## Key Insights

1. **Story AC field paths should be verified against the pinned SDK types, not assumed from docs or training data.** The dahlia drift was caught by the Zod schema (which is good) but could have been caught at story-drafting time by grepping `node_modules/stripe/types/*.d.ts` (which is better). This applies to any story that references third-party API shapes.
2. **Delegate to Stripe what Stripe already solves.** The Stripe best-practices consultation revealed that Stories 5.2 (Payment History), 5.3 (Upgrade), and 5.4 (Cancel) are all handled natively by Stripe's Customer Portal. Building custom UI for these is unnecessary complexity. **This discovery saves ~3 stories of development effort in Epic 5.**
3. **Per-story smoke tests build confidence but don't replace full lifecycle verification.** Root is 80% confident in Epic 4's payment handling. The remaining 20% requires a guided end-to-end walkthrough of the complete subscription lifecycle (signup → subscribe → renew → fail → fix → upgrade → cancel) — which belongs in a dedicated launch-readiness epic.
4. **Deferred production-readiness items should be planned as a launch epic, not carried as retro debt indefinitely.** Items #1, #2, and #4 from Epic 3 were carried for two epics without resolution. Moving them into a planned launch epic gives them first-class priority and a clear timeline instead of perpetual deferral.
5. **No real-time Firestore listeners needed on the dashboard.** Root's direction: fetch `users/{uid}` on load, validate current status server-side before any mutation. This avoids `onSnapshot` complexity and is simpler to build, test, and reason about.

## Team Agreements

- **Delegate to Stripe what Stripe already handles.** Use the Customer Portal for subscription management (payment history, plan changes, cancellation, payment method updates). Build custom UI only for app-specific needs (dashboard overview, status badges, warning banners).
- **No real-time Firestore listeners.** Fetch on load, validate server-side before mutations. Stale reads for a few seconds are acceptable; stale writes are not.
- **Verify SDK type shapes before writing ACs.** Grep `node_modules/<pkg>/types/*.d.ts` for any story that references third-party API field paths.
- **Manual smoke steps must include "where to look."** Every verification step names the terminal, console, or UI element and the exact string/field to check.
- **Carry forward all Epic 3 team agreements** that still apply: no silent `catch {}` in API routes, Human Prerequisites as checkboxes, never paste secrets, `npm run build` in DoD, Single-Source-of-Truth pattern.

## Significant Discovery: Epic 5 Scope Reduction

**Discovery:** Stripe best-practices consultation during this retrospective revealed that the Customer Portal natively handles payment history, plan switching (with proration), cancellation (immediate and end-of-period), and payment method updates.

**Impact:** Epic 5's current 5-story plan includes 3 stories (5.2, 5.3, 5.4) that build custom UI for functionality the portal already provides. Building custom UI is unnecessary complexity, increases maintenance burden, and is explicitly advised against by Stripe's own guidance.

**Resolution:** Revise Epic 5 from 5 stories to ~3 stories. Delegate subscription management to the portal. Build custom UI only for the dashboard overview (reads `users/{uid}`) and payment failure warning banners (app-specific UX the portal cannot push).

**Epic update required:** YES — revise Epic 5 definition before creating stories.
