# Tech-Spec: Hash Accuracy Validator (Tool 4)

**Created:** 2026-03-19

## Overview

### Problem Statement

`map_config.json` holds reference dhashes for 10 maps, but the labeled dataset (`output/labeled/`) has 14 maps with start/end/score frames that keep growing. There is no tool to validate that the chosen hash configuration (dhash, 720p, `map_name_hud` ROI, shift_tolerance=2) correctly identifies each labeled frame — so coverage cannot be confirmed and regressions cannot be caught as frames accumulate.

### Solution

New `tools/hash_validator.py` that loads `map_config.json`, iterates all `*_start.png` and `*_end.png` labeled frames, predicts the map for each frame via nearest hash (shift-tolerant Hamming distance), and reports per-frame and per-map accuracy. Maps with no reference hash in `map_config.json` are listed separately as "no_reference". Outputs `output/accuracy_report.json`.

### Scope

**In Scope:**
- `tools/hash_validator.py` (new file)
- Reuse `build_canvases()` and `hamming_shift_tolerant()` from `hash_comparator.py` (no changes to that file)
- `--shift-tolerance` CLI arg (default: 2)
- `--images` and `--map-config` CLI args
- Per-frame prediction + correctness, per-map accuracy %, overall accuracy %
- Maps missing from map_config → "no_reference" section in report

**Out of Scope:**
- Modifying `hash_comparator.py` or `map_config.json` format
- Score frames
- Any new hash method testing

## Context for Development

### Codebase Patterns

- Modular CLI pipeline: each tool is a standalone script in `tools/` using `argparse`
- `sys.path.insert(0, os.path.abspath(os.path.join(os.path.dirname(__file__), "..")))` at top of every tool for `utils/` access
- ROI coordinates defined at reference resolution 1920×1080, scaled at runtime via `utils/image.scale_roi()`
- Config for Tool 4 comes from `map_config.json` (self-contained: roi, canvas_size, hash_size, hash_method, tile_cols, maps) — NOT from config.yaml
- Frame ground truth is folder name (`output/labeled/<map_name>/`); frame type is determined by filename suffix (`_start`, `_end`, `_score`)
- Output is JSON written to the `output/` directory

### Files to Reference

| File | Purpose |
| ---- | ------- |
| `tools/hash_comparator.py` | Source of reusable functions: `build_canvases()`, `hamming_shift_tolerant()`, `tile_into_canvas()`, `IMAGE_EXTENSIONS`, `compute_hash()` |
| `output/map_config.json` | Input: roi, canvas_size, hash_size, hash_method, tile_cols, reference hashes per map |
| `output/labeled/` | Input: ground truth labeled frames — `<map_name>/*_start.png` and `<map_name>/*_end.png` |

| `utils/config.py` | Not needed — Tool 4 reads map_config.json directly, not config.yaml |

### Technical Decisions

- **Self-contained via map_config.json**: All hash parameters (roi, method, canvas_size, hash_size, tile_cols) are read from `map_config.json`, not `config.yaml`. This mirrors what the mobile app will do and ensures consistency.
- **Reuse hash_comparator functions, not import**: Copy `build_canvases()`, `hamming_shift_tolerant()`, `tile_into_canvas()`, `compute_hash()`, `IMAGE_EXTENSIONS` directly into `hash_validator.py` (or import from the module). Preferred: import from `tools.hash_comparator` to avoid duplication, using `sys.path.insert` for the `tools/` directory.
- **Frame filtering**: Unlike `hash_comparator.load_all_frames()` which loads all images, Tool 4 must filter by suffix: include only filenames ending in `_start.png` or `_end.png`, exclude `_score.png`.
- **Maps with no reference**: Folders in `labeled/` whose name is not in `map_config["maps"]` are reported under `"no_reference"` in the output JSON, not included in accuracy calculations.
- **shift_tolerance**: CLI arg `--shift-tolerance` with default `2` (matching `hash_comparison_report.json`). Not in `map_config.json`, so explicit default needed.
- **text_anchor_width**: Not stored in `map_config.json`, so disabled in Tool 4 (pass `None`). Consistent with how `map_config.json` was generated by `hash_comparator.py` (tiling was active, anchoring was not).
- **Prediction logic**: For each frame, compute hash, then find the map in `map_config["maps"]` with the minimum `hamming_shift_tolerant` distance. Tie-break alphabetically. Predict "unknown" if all maps have distance above a threshold (optional — can omit for V1 simplicity).
- **Output structure**: `accuracy_report.json` with `summary` (overall accuracy, frame counts), `by_map` (per-map accuracy + frame list), `no_reference` (maps skipped due to no hash in map_config).

## Additional Context

### Dependencies

- `imagehash>=4.2`, `opencv-python>=4.8`, `numpy`, `pillow` — already present in the project environment
- `tools/hash_comparator.py` must exist — `build_canvases()`, `hamming_shift_tolerant()`, `compute_hash()`, `IMAGE_EXTENSIONS` are imported from it
- `output/map_config.json` must exist — generated by running `tools/hash_comparator.py` first
- `output/labeled/` must exist — generated by `tools/frame_labeler.py`

### Testing Strategy

- Manual run: `python tools/hash_validator.py` with defaults, inspect `output/accuracy_report.json`
- Verify score frames excluded: confirm `*_score.png` filenames are absent from the report
- Verify no_reference: confirm `bastion`, `coliseum`, `lunar_outpost`, `the_rock` appear in `no_reference`
- Shift tolerance check: run with `--shift-tolerance 0` and compare results to `--shift-tolerance 2` — if accuracy drops, shift tolerance is working as intended
- Regression use: run after adding new labeled frames to confirm accuracy stays at 100% for known maps

### Notes

- `map_config.json` does not store the processing resolution (720p). If `hash_comparator.py` is ever re-run at a different resolution and `map_config.json` is regenerated, `hash_validator.py` must be called with the matching `--resolution`. Consider adding `resolution` to `map_config.json` format in a future update (requires a change to `generate_map_config()` in `hash_comparator.py`).
- When videos for `bastion`, `coliseum`, `lunar_outpost`, and `the_rock` are available and labeled, re-run `hash_comparator.py` to regenerate `map_config.json` with all 15 maps; the no_reference list will then shrink.
- Future: `--fail-on-incorrect` flag could be added to exit with code 1 if any frame is misidentified, enabling CI pipeline use.
